# Outreach

**Status:** Policy (Layer 1). Read by `.claude/skills/outreach-draft-and-send/SKILL.md`.
**Owner:** Outbound Signal OS.

## Purpose

Turn a research dossier into personalized outreach drafts across the channels that fit this lead, then hold every draft in an approval queue. Nothing generated by this stage sends itself.

## Drafting rules

- Every claim in a draft must trace to the dossier's "safe claims" list (`research.md`). Never assert something the dossier only implied.
- Not every lead gets every channel. Draft the channel(s) the dossier's route recommendation actually supports: offering a resource, a LinkedIn touch, an email, or a WhatsApp message only where the evidence and the lead's public presence make it a fit.
- Each touch in a sequence adds new value. A follow-up that just repeats the first message with different framing ("bumping this") is not a real follow-up.

## Approval gate (non-negotiable)

**Every send requires explicit human approval before it happens.** Drafts are created, staged, and queued: never sent automatically, regardless of how confident the scoring or research stages were upstream. This is the same rule the real production system runs on, not a demo simplification.

## Channel notes

- **LinkedIn / email**: primary channels for most leads; can carry a screenshot, a proof link, or a resource offer alongside the message text.
- **WhatsApp**: used sparingly, mainly where a prior touch established some relationship or the lead's context makes it appropriate; never the first-touch channel by default.
- **Resource offer**: a short, specific, useful artifact (not a generic PDF) offered only when the dossier genuinely supports one being useful to this lead; skipped when it would read as homework.

## What this directive does not cover

How a reply gets classified and routed once it comes back (`reply_handling.md`), or how variant copy gets tested against each other (`learning_loop.md`).
